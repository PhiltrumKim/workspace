# Magewell RPA 수정 내역 (2026-07-05)

릴레이보드(TX-RelayBoard) 연동 마우스/키보드 제어 및 OOP GUI(`run_oo.py`)의
버그 수정·기능 개선 기록. 실행 환경은 프로젝트 `.venv` (Python 3.9, serial 3.5, opencv 4.13).
대상 하드웨어: COM3 = Silicon Labs CP210x(릴레이보드), 캡처 = Magewell Pro Capture DVI(1920×1080),
제어 대상 = Raspberry Pi 데스크톱.

## 1. 프로토콜 정합성 — 마우스 Δy 부호비트 (`core/relay_board.py`)

### 배경
프로토콜 문서 `[20250310]TX_RelayBoard_Protocol.pptx`(p.8, Mouse Format 2.0)의 부호비트 규칙은
"패킷 저장값(#7~#10)이 음수일 때 해당 부호비트 set"으로 일관된다.
- Δx 저장값 = `dx`      → `dx < 0` 일 때 bit4(0x10)
- Δy 저장값 = `-dy`(반전) → `dy > 0` 일 때 bit5(0x20)  (문서 표기: `001x 1xxx : delta y > 0`)

기존 코드는 `if delta_y < 0: sign_bits |= 0x20` 으로 **문서와 반대**로 bit5를 설정하고 있었다.
(코드 주석은 올바르게 적혀 있었으나 구현만 반대)

### 하드웨어 검증
COM3 실기로 current(#6=0x28)/spec(#6=0x08) 두 방식을 동일 Δy로 전송한 결과, **커서가 동일하게 이동**.
→ **펌웨어는 부호비트(#6 bit4/bit5)를 사용하지 않고, #7~#10의 signed int16 값만으로 이동을 결정**한다.
따라서 이 불일치는 실동작에 영향이 없으나, 문서 정합성을 위해 수정.

### 변경
- `send_mouse_move()`의 조건을 `if delta_y < 0` → **`if delta_y > 0`** 로 수정
- 상단 docstring의 X/Y 비트 위치 표기 정정(bit4=Δx, bit5=Δy) 및 "펌웨어는 부호비트 미사용" 주석 추가
- 5개 케이스 재검증: 문서 기대값과 모두 일치, 실이동 동일

## 2. GUI 버그 수정 (`ui/main_window.py`)

### 2-1. 수동 ROI 설정 크래시
- **증상**: "수동 ROI 설정(r)" 실행 시 프로세스 강제 종료
  (`Fatal Python error: PyEval_RestoreThread ... GIL`)
- **원인**: Tkinter `mainloop` 안에서 `cv2.selectROI`(OpenCV HighGUI가 자체 이벤트 루프 구동) 호출 → GIL 충돌
- **수정**: cv2 제거, **Tkinter 네이티브 드래그 방식**으로 교체
  - `roi_mode` 플래그 + `video_label`에 `<ButtonPress-1>/<B1-Motion>/<ButtonRelease-1>`, `<Escape>`(취소) 바인딩
  - 드래그 중 초록 미리보기 박스를 `update_frame`에서 렌더링
  - 프레임은 원본 해상도로 표시되므로 라벨 좌표 = 프레임 픽셀 좌표(1:1)

### 2-2. 추종 마우스 폭주 → UI 프리즈
- **증상**: 릴레이 연결 + 템플릿 매칭 시 프로그램이 멈춘 것처럼 보임(로그 573KB 폭주)
- **원인**:
  1. `update_frame`(메인루프) 안에서 `move_mouse_relative`가 큰 delta를 127씩 쪼개 **동기 전송**
     (각 `send_packet`에 `time.sleep(0.01)`) → 메인루프 블로킹
  2. 시각 피드백 루프(마우스 이동 → 캡처 영상 변화 → 재매칭 → 재이동) 폭주
  3. 매 프레임 `[Track Debug]` + 컨트롤러 `[TX]` print 폭주
- **수정**:
  - 컨트롤러 `verbose=False` (연결 시)
  - 기존 디버그 print 제거
  - 추종 이동을 **프레임당 1패킷(±127)로 제한 + 데드존 2px**, `send_mouse_move` 단발 전송
    (잔여 오차는 다음 프레임에서 재수렴 → 비블로킹)

### 2-3. 좌표 오프셋 자동 보정 (~46px)
- **증상**: 커서가 템플릿보다 X·Y 모두 일정하게 ~46px 벗어남(양축 동일 상수 오프셋)
- **원인**: 캡처 프레임의 검은 여백/오버스캔. `reset_mouse`가 커서를 실제 화면 (0,0)으로
  홈잉하지만 그 지점이 캡처 프레임에서는 (~48,48)이며, 앱은 이를 `current=(0,0)`으로 간주 → 전 좌표가 밀림
- **수정**: `toggle_relay_connection`에 **`_calibrate_home_origin(controller)`** 추가
  - `reset_mouse` 후 (+8,+8) jiggle-diff로 캡처 프레임상 실제 커서 위치를 측정해 `current` 초기값에 반영
  - 오버스캔/해상도 변화에도 자동 대응
  - `import time` 추가
- **결과**: Menu.png 도달 오차 (+47,+46) → **(+6,+6)** 로 개선

### 2-4. 클릭 정확도 — 중심 타깃
- **변경**: `update_frame`의 target 계산을 검출 박스 top-left → **중심**으로 변경
  ```python
  center_fx = x + w / 2.0
  center_fy = y + h / 2.0
  target_x = int((center_fx / frame_w) * monitor_w)
  target_y = int((center_fy / frame_h) * monitor_h)
  ```
- **효과**: 커서 팁이 템플릿 중앙에 위치 → 버튼 클릭에 적합
- **검증**: Menu 버튼 중심(1084,67) 도달, 오차 dx6/dy7

### 2-5. 진단 로그 (유지)
- `update_frame`에 15프레임마다 1회(스로틀) `[Track]` 로그 출력
  (frame 크기, 매칭 score, target, cur, delta). 폭주 없음.

## 3. 검증 방법 및 결과

Magewell 캡처를 cv2(index 0, 1920×1080)로 직접 잡아 헤드리스로 구동/측정.
Pi 화면 절전 시 릴레이 마우스 흔들기로 깨움.

| 항목 | 결과 |
|------|------|
| 하드웨어 X 응답 | 명령 (+500,0) → 실측 (+500, 0) : 1:1 정확 |
| 하드웨어 Y 응답 | 명령 (0,+500) → 실측 (+1, +505) : 방향·크기 정확 |
| 앱 추종 수렴 | 홈(0,0)에서 시작, 목표 X·Y 동시 정확 수렴 |
| 상수 오프셋(보정 전) | +47, +46 (양축 동일 = Y 고유 문제 아님) |
| 자동 보정 후 오차 | +6, +6 (하드웨어 이동 정밀도 ±1% 수준) |
| 중심 타깃 후 | Menu 버튼 중앙 ±7px 도달 |

**결론**: 사용자가 관찰한 "Y좌표 이상"은 Y축 고유 버그가 아니라,
① 수정 전 추종 폭주/프리즈 상태의 잔상, ② 양축 공통 오프셋(오버스캔)이 원인이었으며 모두 해소됨.

## 5. 알려진 잔여 사항 / 향후

- 잔여 도달 오차 ~6~7px: 릴레이+Pi 상대이동의 물리 정밀도(±1%) 및 측정 노이즈 수준. 실사용에 충분.
- 명세 대비 미구현 커맨드(선택): 초기화 시퀀스(SW리셋→버전확인→KM config), 버전확인(0xFD+#7=0x01),
  Alive ack(0xE2), PCC(0xE4), 전체리셋(0xE3).
- `reset_mouse`는 문서상 0xE0(#7~#10 해상도 지정)을 쓰지 않고 (-127,-127)×35 SW 호밍으로 우회(의도적).
